# План приложения «Семейные рецепты»

Приложение для хранения рецептов: девушка загружает блюда, мы быстро выбираем что
готовить на завтрак/обед/ужин и упрощаем походы в магазин.

## Зафиксированные решения

- **Вход:** общий код на семью — все члены входят по *названию семьи + секретному коду*
  и видят общие рецепты и календарь.
- **Список покупок:** делаем во второй версии (Этап 3) — авто-сбор продуктов из блюд,
  запланированных в календаре на выбранный период, в единый список с галочками.

## 1. Ключевые фичи

- Каталог рецептов с карточками.
- Главная = список рецептов + фильтры: приём пищи, примерное время готовки,
  поиск по названию, фильтр по тегам (включая «мясное» — это обычный тег).
- Календарь: к дате прикрепляем одно или несколько блюд.
- «Семья»: вход по названию + коду; у каждой семьи свои рецепты и календарь.
- Для создания блюда достаточно **только названия**, всё остальное опционально
  (фото, описание, продукты, теги).

## 2. Карточка блюда

- Фотография (опц.)
- Название (**обязательно**)
- Описание — рецепт и заметки (опц.)
- Список продуктов в формате «что и сколько»: `сыр — 200 гр`, `соль — по вкусу` (опц.)
- Теги: `#обед` `#ужин` `#мясное` и любые свои (опц.)

## 3. Технологический стек и хостинг (всё бесплатно, настройка из терминала)

| Слой | Решение | Терминал |
|------|---------|----------|
| Фронтенд | Next.js (React) + TypeScript + Tailwind | `npx create-next-app` |
| Хостинг приложения | Vercel (Hobby, бесплатно для личного) | `vercel` CLI |
| БД + хранилище фото | Supabase (free: Postgres + Storage) | `supabase` CLI |

Пару начальных шагов (создание проекта, ключи) делаем в веб-консоли один раз; схема БД и
деплой — из терминала. При настройке подтянуть актуальную официальную документацию
(Context7), чтобы команды и лимиты были точными.

## Архитектура проекта (FSD)

Feature-Sliced Design. Слои (импорт только «сверху вниз»):

- `app/` — роутер Next.js + app-слой (root layout, провайдеры, `globals.css`).
- `views/` — экраны (FSD-слой «pages», переименован, т.к. `pages` зарезервирован Next).
- `widgets/` — композитные блоки (`app-shell`: шапка + нижняя навигация).
- `features/` — действия пользователя (`filter-recipes`, `edit-recipe`).
- `entities/` — бизнес-сущности (`recipe`: типы, данные, карточка).
- `shared/` — переиспользуемые примитивы (`ui`: кнопки, чипы, поля, иконки).

Роут-файлы Next тонкие: импортируют экран из `views` и пробрасывают параметры.
У каждого слайса публичный API через `index.ts`. Алиас `@/*` → `src/*`.

## 4. Модель данных

```
families      id, name, code_hash, created_at
recipes       id, family_id, name (обязательно), description, photo_path,
              cook_time_minutes, meal_types[], tags[], created_at, updated_at
ingredients   id, recipe_id, name ("сыр"), amount ("200 гр"), sort_order
meal_plan     id, family_id, date, recipe_id, meal_slot, created_at
```

- Поля для фильтров (приём пищи, время) — структурные, чтобы фильтры были надёжны.
  «Мясное» — обычный тег; фильтровать можно и по тегам.
- `tags[]` — свободные дополнительные метки.
- Продукты — отдельной таблицей (нужно для будущего списка покупок).
- `amount` — текст, чтобы влезали «по вкусу», «2 шт».

## 5. Безопасность («семья»)

- Клиент НЕ обращается в базу напрямую. Все запросы идут через серверную часть Next.js
  (на Vercel); секретный ключ Supabase живёт только на сервере, в браузер не попадает.
- Вход: название + код → сервер сверяет код с **хешем** (сырой код не хранится) → выдаёт
  подписанную сессию (HttpOnly cookie с `family_id`).
- Все чтения/записи привязаны к `family_id` из сессии → чужие данные недоступны.
- Код длинный и случайный (8–10 символов); на эндпоинт входа — ограничение попыток.
- Фото в приватном бакете (`family_<id>/...`), отдаются по временным подписанным ссылкам.
- Дополнительно включаем RLS в Postgres как второй рубеж.

## 6. Экраны

1. Вход / создание семьи (название + код)
2. Главная — сетка карточек + фильтры и поиск
3. Карточка блюда
4. Создание/редактирование (обязательно только название)
5. Календарь (месяц/неделя; тап по дню → прикрепить блюда)
6. Список покупок (Этап 3)
7. Настройки (имя семьи, код, выход)

## 7. Дизайн и палитра (без градиентов)

Работаем строго в четырёх цветах (плюс их прозрачности для границ/теней/неактивных
состояний — это не новые цвета):

- `#F9F6F1` — основной тёплый фон
- `#181B17` — текст, иконки
- `#D07E42` (терракота) — главный акцент: ключевые кнопки, активные элементы, «сегодня»
- `#B7AD76` (приглушённое золото) — вторичный акцент: теги, рамки, второстепенные кнопки

Только плоские заливки, аккуратные рамки, скруглённые карточки, много воздуха.

## 8. Бесплатные лимиты — подводные камни

- Supabase free (подтверждено по офиц. докам): проект **ставится на паузу после 1 недели
  без активности**; до 2 активных бесплатных проектов одновременно (на паузе не считаются).
  ~500 МБ БД, ~1 ГБ хранилища.
- Фото сжимаем на клиенте перед загрузкой (~1280px) — останемся в пределах гигабайта.
- Vercel Hobby — бесплатно для личного некоммерческого использования.

### Keep-alive (защита от паузы Supabase)

Паузу сбрасывает **реальная активность БД**, а не открытие статической страницы. План:

1. Health-эндпоинт `GET /api/health` в Next.js, делающий копеечный запрос к базе
   (`select 1`) через серверный клиент Supabase. Защищён секретным заголовком.
2. Расписание — **Vercel Cron** в `vercel.json`, раз в 2 дня (запас до недельного порога):
   ```json
   { "crons": [{ "path": "/api/health", "schedule": "0 9 */2 * *" }] }
   ```
3. Запасные варианты: GitHub Actions cron-workflow либо внешний пингер
   (cron-job.org / UptimeRobot) по тому же URL.

Точную допустимую частоту Vercel Cron на Hobby сверить по актуальным докам при настройке.

## 9. Этапы реализации

- **Этап 0** — каркас Next.js, тема/палитра, пустой деплой на Vercel (проверка пайплайна).
- **Этап 1 (MVP)** — Supabase, схема, серверная авторизация семьи, CRUD рецептов с фото,
  список + фильтры, карточка. Здесь же — `/api/health` + Vercel Cron для keep-alive.
- **Этап 2** — календарь.
- **Этап 3** — список покупок (авто-сбор продуктов из календаря).
- **Этап 4** — полировка + PWA (установка на телефон, иконка, опц. офлайн).

## Известные проблемы / доработки

- **Визуальный баг (нижняя навигация):** при переключении «Рецепты» → «Календарь»/
  «Настройки» меняется высота нижнего меню (оно становится выше). Вероятно из-за
  появления/исчезновения вертикальной полосы прокрутки на страницах разной высоты или
  отсутствия фиксированной высоты у `BottomNav`. Зафиксировать высоту меню. Поправить
  при UI-полировке (Этап 4) или раньше.
- **Сброс/просмотр кода семьи:** реализовано (просмотр и регенерация в настройках).
- **Rate-limiting на вход по коду** — добавить позже (защита от перебора).
